# Subtasks and Orchestrate

Two ways to delegate work to another LLM invocation while keeping the main chat clean.

## subtask(task, context?, persona?)

Spawns a **fresh** LLM conversation with the same toolset but **no memory** and **no RAG**.
Runs a ReAct-style loop: the sub-LLM thinks, uses tools, and returns a single final answer.

- Use for exploratory/analytical work whose intermediate reasoning would pollute the main
  chat: "find TODOs in src/", "analyze this log", "research and call the right external
  tool for X".
- `context` is optional — pass any background facts the sub-task needs. The sub-task does
  NOT see your chat history; everything must be in `task` or `context`.
- `persona` optionally overrides the default "task executor" system prompt.
- Base settings (from `set "..."`) are always included automatically.
- Hard iteration limit (`ai.subtask.max-iterations`, default 10) — the sub-task aborts if
  it keeps calling tools without reaching a final answer.

Only the final answer is returned to the main chat; the sub-LLM's tool calls and
intermediate thoughts are not visible.

## orchestrate(problem)

Runs a structured multi-agent pipeline generated by a separate planner LLM. The planner
picks 2-5 specialist agents (plus optional `forEach` loops) that run sequentially, with
intermediate outputs optionally fed into RAG. The final agent's output is returned.

- Use for structured deliverables: "write a 5-chapter novel", "compare three approaches",
  "research + outline + draft + review".
- Uses `FORCE_SYSTEM` planner prompt — no questions asked, sensible assumptions made.
- Intermediate outputs are printed live to the console but are NOT returned to the main
  chat; only the final agent's result.

## Which to pick

| Situation | Tool |
|---|---|
| Ad-hoc "go figure this out" | `subtask` |
| "Write X in multiple distinct phases" | `orchestrate` |
| Need Tools + multi-step reasoning | `subtask` |
| Structured deliverable with review step | `orchestrate` |
| Unsure | `subtask` is the safer default |

## Nesting rules

- `subtask` can call `orchestrate` (pipeline inside a sub-task).
- `subtask` cannot call `subtask` (prevents unbounded recursion).
- `orchestrate`'s sub-agents have access to all `AgentTool`s including `orchestrate` (rare
  but possible — their own iteration limit will stop runaway loops).
